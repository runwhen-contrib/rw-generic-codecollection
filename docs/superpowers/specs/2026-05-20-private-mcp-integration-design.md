**Status:** Design proposal — engineering review
**Date:** 2026-05-21
**Author:** Prathamesh Lohakare (with Claude)
**Decision owner:** Platform team

---

## Executive Summary

**Problem.** RunWhen customers run private MCP servers inside their own VPCs. The hosted platform — specifically the agentfarm chat/agent runtime — needs to invoke tools on those servers as part of agentic chat sessions. The wire constraint is non-negotiable: the platform cannot reach into a customer VPC; only the in-VPC runner can speak outbound to the platform.

**Recommendation.** **Approach D — 1 SLX per MCP tool (tools-as-tasks).** Every MCP tool becomes a first-class RunWhen task. Per-tool SLXs are auto-generated by the existing runner-side workspace builder using a standard generation rule + templates (the same pattern every other codebundle in the collection already uses). agentfarm needs **zero new code** — MCP tools are just more tasks for the existing task-search / task-run workflow.

**Why D wins (vs. the alternatives we considered):**
- **Smallest implementation surface of the four approaches** — codebundle + generation rule + small papi CRUD + UI form. Weeks of work, not months.
- **Zero agentfarm changes** — the agent already knows how to search workspace tasks and run them.
- **No new sync infrastructure** — the workspace-builder cycle is the sync, reusing the same pattern that already keeps namespace / deployment / function SLXs in sync.
- **Unified LLM surface** — one mental model for the agent (everything is a task), not "search for tasks vs. pick from MCP-tools-in-context."
- **MCP tools as first-class platform entities** — per-tool RBAC, audit, search, scheduling, runbook composition all extend for free.

**Trade-offs accepted:**
- **Scope is tool calls only.** Resources / prompts / sampling / streaming progress are explicitly out of scope. If those become roadmap requirements, revisit Approach B (gateway).
- **Latency floor is SLX-run.** ~1–3 seconds hot path with priority worker class + warm caches; up to ~15s cold. Acceptable for chat-loop tool calls (people wait that long for LLM tokens anyway); not suitable for ~100+ calls/min/customer workloads. If volume rises that high, revisit Approach B.
- **More SLXs per workspace** (N servers × M tools each). Mitigated by tagging / grouping in the UI.

**One decision the team needs to land before implementation:** how the workspace-builder learns about registered MCP servers — pull from papi via outbound API (Option D1) vs. read from runner Helm values (Option D2). See §7.4.

**Out of scope for this design (explicitly):** MCP resources/prompts/sampling/streaming/cancellation, MCP transports other than HTTP/JSON-RPC, per-MCP-server RBAC at the workspace level, billing/metering, MCP-server auto-discovery in the customer cluster. See §11.

## 1. Background & Goal

RunWhen customers run **private MCP servers** inside their own VPCs (e.g. a self-hosted Atlassian/Jira MCP, an internal observability MCP, a custom infra MCP). RunWhen's hosted platform — specifically the **agentfarm** chat/agent runtime (Google ADK based) — needs to invoke tools on those private MCP servers as part of agentic chat sessions.

The wire constraint is the load-bearing thing: **the platform cannot reach into the customer's VPC directly**. Inbound connectivity is what private-VPC customers are paranoid about. The runner already lives inside the VPC and maintains an outbound channel to the platform, so any solution must route through that VPC ↔ platform direction.

**Scope for this design:** *tool calls only*. Resources, prompts, sampling, server-initiated requests, streaming progress, and other MCP surface area are **out of scope for the near-term roadmap** but factor into long-term extensibility judgement.

## 2. Constraints & Assumptions

- **Network:** outbound-only from VPC → platform. No inbound holes punched into customer infra.
- **Latency target:** ideally sub-second per tool call; both approaches will have polling overhead, so realistic target is "a few seconds". A priority-class worker on the runner side is mentioned as a mitigation that applies to either approach.
- **Consumer:** agentfarm is the primary near-term consumer. Other consumers (UI, automation runbooks) may follow but aren't blocking.
- **MCP scope:** tool calls only.
- **Auth/secrets:** must reuse the existing runner-side vault/secret-management plumbing — no new credential systems.
- **Customer ops surface:** prefer zero or one new deployable component over many.

## 3. Today's baseline (briefly)

- One `Runbook` row in the platform DB carries `env_vars` + `secrets` shared by every task in the .robot file.
- Multi-task .robot files run as **one `robot.run()` invocation** (see `platform-robot-runtime/build-artifacts/runrobot.py:848-861`) with a `test=titles` filter selecting the subset (driven by the `RW_TASK_TITLES` env var).
- Runtime variables already exist — papi merges them into config vars at run time. The robot file never sees the distinction.
- The runner is in the VPC and polls the platform outbound for queued task work. Output (issues, metrics) flows back the same way via `run_output.json` and `RW.Core.Add Issue` / metric pushes.
- The current `codebundles/tool-builder/runbook.robot` supports exactly **one task per SLX** — title, env, secrets, script are all singletons.

**Four approaches are explored** in this doc:
- **A — Multi-task SLX** (per-task config materialized in Runbook)
- **B — MCP Gateway Proxy** (in-VPC gateway speaks native MCP end-to-end)
- **C — 1 SLX per MCP server, tools-as-context** (LLM sees tools list in chat context)
- **D — 1 SLX per MCP tool, tools-as-tasks** (every MCP tool becomes a first-class RunWhen task)

A and C share the SLX-as-proxy model but differ in granularity (server-level vs. tool-level) and consumer integration (in-context vs. first-class task). D is the most fully integrated; B is the most protocol-faithful. The recommendation lands on D — see §9.

## 4. Approach A — SLX-as-MCP (tasks-as-tools)

Wrap each private MCP server as an SLX whose tasks correspond 1:1 to the server's MCP tools. Agentfarm invokes a task via papi's task-run API, the runner picks up the SLX run, the codebundle calls the local MCP server inside the VPC, and the result flows back through the existing output channel.

### 4.2 Data flow (per tool call)

1. Agentfarm decides to call tool `create_jira_issue` on the customer's Jira MCP SLX.
2. Agentfarm invokes the task through the existing task-invocation flow (the same path it uses for any RunWhen task), passing `args={...}` as runtime variables.
3. papi validates `args` against the stored per-task input schema, enqueues a task work item via existing machinery.
4. Runner picks up the work via existing outbound poll, runs the SLX via `runrobot.py`.
5. The codebundle (a generic MCP-call wrapper script): reads `RW_INVOCATION_ARGS_JSON`, makes the HTTP call to the local MCP `tools/call` endpoint, writes the result to `run_output.json`.
6. Output flows back through the standard runner → platform channel.
7. papi correlates the run result with agentfarm's pending request, returns the tool output.

### 4.3 Sub-variants

**A1 — For-loop in runbook.robot** (recommended within this approach)
- One task body wrapping `FOR ${t} IN @{tasks_subset}: Run Tool Builder Task ${t}`.
- Platform filters the subset upstream; robot just iterates whatever it receives.
- No `RW_TASK_TITLES` reliance, no slot ceiling, no cross-repo runner change.
- **Cost:** Robot sees one task per run. Native per-task observability (`output.xml`, log.html, per-task timing) collapses into one entry. For MCP wrapping specifically this doesn't matter — agentfarm invokes one tool at a time so subset is always size 1 — but for "batch multi-task runs" it does.

**A2 — Prerunmodifier (Robot SuiteVisitor)**
- Codebundle ships a `task_generator.py` modifier that reads `TASKS_JSON` and synthesizes N first-class Robot tasks in memory before execution.
- Requires `runrobot.py` (in `platform-robot-runtime`) to expose a `RW_PRERUNMODIFIER` env var and pass it through to `robot.run(prerunmodifier=...)`. **This is a cross-repo change** with coordinated release.
- Preserves Robot-native per-task observability.

Pre-declared slot tasks (an earlier idea) was discarded because for-loop is strictly simpler with the same trade-offs.

| Repo | Change |
|---|---|
| `rw-generic-codecollection` | Rewrite `codebundles/tool-builder/runbook.robot` to for-loop over `tasks_subset` (A1) or remove `*** Tasks ***` entirely + ship `task_generator.py` (A2). The task body becomes a generic MCP HTTP client (or stays as user-provided gen_cmd for non-MCP tool-builder use). |
| `papi` | (a) Extend `Runbook` row to carry a `tasks: [...]` blob (or new `Task` entity) with per-task `{title, description, input_schema, gen_cmd, config_env, secret_env}`. (b) New `config_shape: "shared" \| "per_task"` flag on Runbook for consumer-side branching. (c) MCP introspection service: call upstream `tools/list`, materialize SLX, cron resync. (d) Schema validation of invocation args. **Note:** task invocation and output reading reuse existing endpoints — no new task-run API needed. |
| `agentfarm` | **None.** Agentfarm already runs tasks and reads their outputs. Per-task description / input_schema are exposed via the existing task-metadata surface (consumed via the `config_shape: "per_task"` branch). |
| `runwhen-runner` | None for A1. For A2, none directly — change is in `platform-robot-runtime`. |

### 4.5 Pros

- **Reuses everything.** Runner, secrets, output channel, audit, observability, scheduling. No new deployable.
- **Outbound-only networking preserved** — exact same network shape as today.
- **Auditable as task runs** — every tool invocation is a first-class SLX run in the platform UI, replayable, traceable.
- **Customer doesn't deploy anything new.**

### 4.6 Cons / risks

- **Per-task data model is a platform schema change.** The Runbook today carries shared env/secrets. Multi-task needs per-task config — either as a blob inside Runbook (cheaper) or a new `Task` entity (more invasive). Either way, **every consumer of the Runbook API gets the multi-task case** and must branch on `config_shape`.
- **Schema drift.** MCP servers can change `tools/list` at any time. SLX is materialized once; without a re-sync mechanism, the SLX goes stale. A cron job is the obvious fix but introduces yet another moving part — pick the resync strategy deliberately (on-demand reintrospection per call vs. periodic cron vs. webhook).
- **MCP semantics ceiling.** Only tool calls fit cleanly. If resources / prompts / sampling enter the roadmap, this path runs out of runway and the gateway becomes the right answer anyway. SLX investment becomes partial waste.
- **Latency from full SLX run path.** Worker pickup → robot suite init → secret fetch → script exec. Probably 5-30s per call without a priority worker; less with one but never trivial.

### 4.7 Open questions for A

- **Resync strategy:** on-demand vs. cron vs. webhook? Affects freshness/latency trade-off.
- **Concurrency:** if agentfarm fires N parallel tool calls, can the runner's worker pool fan out N parallel SLX runs against the same SLX? Today's worker pool semantics?
- **Per-task entity vs. blob-in-Runbook:** spec needs to commit to one. Blob is cheaper for this use case; entity is correct if other use cases (per-task RBAC, per-task SLO, per-task issue grouping) are on the roadmap.
- **Name collision on "task":** Robot-task vs. platform-task vs. MCP-tool. Pick a non-confusing name for the new per-task entity (e.g., `tool_entry`, `entrypoint`).

## 5. Approach B — MCP Gateway Proxy

Deploy a small gateway component co-located with the runner in the customer VPC. The gateway maintains an outbound channel to the platform; the platform exposes a native MCP endpoint to agentfarm and dispatches each tool call to the right gateway. **One gateway per workspace** can front **multiple upstream private MCP servers** (Jira, Linear, custom infra, …) — the gateway routes each call to the right upstream by `server_id`. The gateway speaks native MCP to each local server and returns responses through the same outbound channel.

### 5.2 Data flow (per tool call)

1. Agentfarm uses ADK's standard MCP client, configured with `mcp://papi.../workspaces/{ws}/mcp/{server_id}` as one of its MCP servers (one MCP-server registration per upstream — Jira, Linear, etc.).
2. ADK calls `tools/call` over the MCP wire protocol → papi's MCP router endpoint. Payload carries `{server_id, tool_name, args}`.
3. papi router holds the request on the gateway's outbound channel (long-poll or persistent stream — see §5.8).
4. Gateway receives the payload, looks up `server_id` in its local config, dispatches to the right upstream MCP server via its in-process MCP client pool.
5. Local MCP server responds; gateway POSTs the response back over the outbound channel.
6. papi MCP router pairs the response to the original ADK call and returns it.

### 5.3 Required changes by repo

| Repo | Change |
|---|---|
| **NEW repo** (e.g. `runwhen-mcp-gateway`) | Gateway binary. Outbound polling/streaming protocol client. Upstream MCP client pool (one client instance per registered server). Server-id → upstream-URL routing table. Credential bridging (reuse runner's vault integration). Config: which upstream MCP servers to expose. **One gateway instance per workspace** (multi-server, single-tenant). |
| **Deploy charts** | Add gateway as a sidecar or chart add-on co-deployed with the runner. Same lifecycle. |

| `agentfarm` | Configure papi's MCP endpoint as an ADK MCP server. **Zero custom adapter code** — ADK's MCP client handles `tools/list`, `tools/call`, response shaping. |
| `rw-generic-codecollection` | None. |
| `runwhen-runner` | None (gateway is a separate process; co-deploy via chart, not embedded). |
| `platform-robot-runtime` | None. |

### 5.4 Pros

- **Full MCP semantics preserved.** Live schema discovery via `tools/list`, native error model, full input/output types. No drift problem.
- **Zero changes to Runbook/Task data models.** MCP concerns are completely isolated to the new component + the papi router.
- **agentfarm gets the standard ADK MCP integration.** No custom adapter to maintain — every MCP server (public, runwhen-internal, or customer-private) looks the same to the agent.
- **Future extensibility for free.** Resources, prompts, sampling, streaming progress — all map natively when/if they become roadmap items. SLX path can't extend there.
- **No platform schema migration.** Doesn't touch the Runbook model with its many existing consumers.
- **Concurrency is native to MCP.** Multiple in-flight calls per session work the way ADK expects.

### 5.5 Cons / risks

- **New deployable component.** Customer's deploy story gains one more chart/container. Lifecycle, versioning, upgrades, support burden.
- **New outbound protocol** between gateway and papi. Long-poll vs. gRPC streaming vs. SSE — needs design. Not hard but not free; needs to handle reconnects, request expiry, cancellation, backpressure.
- **papi-side router** is non-trivial — request multiplexing, correlation, timeout handling, gateway health.
- **Auth/secrets bridge** has to be built (or carefully shared with the runner's vault integration).
- **Less SLX-style observability.** MCP calls bypass the runbook/issue pipeline. Visibility in the platform UI requires explicit instrumentation (call logs, metrics) on the router side.

### 5.6 Multi-server per gateway (and multi-tenancy)

**Resolved.** The gateway is **per-workspace**. One workspace can register N upstream private MCP servers (Jira, Linear, custom infra, …) — all proxied through the same gateway instance, dispatched by `server_id`. The workspace is the strong isolation boundary.

Why not one gateway per upstream MCP server? Operational simplicity — one gateway container per workspace is enough; the gateway maintains an MCP client pool indexed by `server_id` and routes per-call. Customers don't deploy or version N separate containers.

Why not one gateway shared across workspaces? Trust boundary. A gateway sees plaintext credentials for every upstream MCP server it proxies; cross-workspace sharing would mean every gateway operator has access to every workspace's credentials. Per-workspace keeps that blast radius tight.

From agentfarm's perspective: each upstream MCP server is a separate ADK-MCP-server registration (`mcp://papi/.../mcp/jira`, `mcp://papi/.../mcp/linear`, …) — standard MCP-client semantics, no special multiplexing logic needed on the agent side. The gateway-level fan-out is hidden behind papi's routing.

### 5.7 Why papi in the middle?  (or: could agentfarm talk directly to the gateway?)

Reasonable question. The proposal: agentfarm becomes the rendezvous endpoint that gateways connect to outbound; papi is only involved in setup (discovery + token issuance), not the per-call data path.

#### 5.7.1 Cross-replica routing (applies equally to both designs)

Both papi and agentfarm are horizontally scaled, and both designs have the same structural problem: a gateway's persistent connection lives on **one** replica, but incoming requests from the caller land on (likely a **different**) replica. The pattern is the same in either case:

```
caller → [LB] → Replica P_recv  (request handler holding caller's HTTP response open)
                       │
                       │ 1. write {request_id, gateway_id, payload} to Redis queue keyed by gateway_id
                       │ 2. subscribe to "response:{request_id}" channel
                       v
                  [Redis pub/sub]  ← shared bus, all replicas connected
                       ^
                       │ ↑ G picks up request via its persistent stream
                       │
G ⇄ persistent stream ⇄ Replica P_conn   (this replica holds G's socket)
G → upstream MCP → result
G → POST result → [LB] → Replica P_resp  (any replica)
                              │
                              └─ publishes to "response:{request_id}"
                                 ↓
                            P_recv (subscribed) wakes up, writes HTTP response to caller
```

The mechanism is **Redis pub/sub** (or NATS, or any shared bus). Both papi and agentfarm have Redis access today, so neither side has a meaningful infrastructural advantage here — it's the same plumbing built in whichever service owns the data path.

Also worth being honest about: **papi does not have a long-lived connection management pattern today.** The existing papi ↔ runner protocol is short-lived HTTP in both directions (runner polls for work, runner POSTs on completion; state lives in the DB). So the "persistent bi-directional stream to the gateway" concept is new code regardless of whether we put it in papi or agentfarm.

#### 5.7.2 What actually differs

Given cross-replica routing and connection management are roughly equivalent effort in either service, the real choice is about **where the responsibility lives** — what the service does for the platform, not what infra it already has.

| Concern | Via papi | Direct agentfarm ↔ gateway |
|---|---|---|
| **Reusable by other consumers** (platform UI's "test this MCP tool" button, future agents, automation workflows) | Yes — anything that calls papi can route through | No — every consumer re-integrates with agentfarm, making agentfarm a platform dependency for non-chat use |
| **Auth / RBAC at request time** | Inline FastAPI middleware (existing) | Either inter-service call to papi per request *or* pre-issued capability tokens (new flow either way) |
| **Audit logging** | Free via papi's existing audit pipeline | Gateway or agentfarm has to explicitly push audit events back to papi |
| **Operational role of agentfarm** | Stays a chat-loop service | Promoted to critical platform infra (rendezvous endpoint for gateway connections) — SLA / ops / on-call status change |
| **Hops per call** | agentfarm → papi → gateway → MCP | agentfarm → gateway → MCP (one less hop) |
| **Latency** | One extra papi RTT (sub-100ms in healthy cluster) | Lower by that RTT |

#### 5.7.3 Recommendation

**Via papi for v1.** Honest reasons:

1. **Generalizable.** If MCP-via-gateway is in papi, the platform UI can invoke MCP tools (e.g., a "test this tool" button on a workspace settings page), future agents can use them, and automation workflows can call them. In direct mode, every non-chat consumer either re-implements the gateway protocol or starts depending on agentfarm — and we don't want agentfarm to become a load-bearing platform service.

2. **Inline auth via existing FastAPI middleware.** papi's RBAC checks already wrap every route. Adding MCP routes inherits that for free. In direct mode, agentfarm makes an inter-service auth call per request or operates on pre-issued capability tokens — both add a new flow we don't have today.

3. **agentfarm's operational role stays narrow.** Today agentfarm is a chat-loop service with chat-session SLAs. Direct mode promotes it to "rendezvous endpoint for outbound gateway connections" — a status change that affects on-call rotation, SLA expectations, scaling assumptions, deployment cadence.

4. **Latency cost is manageable.** One papi RTT is sub-100ms in a healthy cluster; upstream MCP RTT (private Jira, Linear, whatever) typically dwarfs it. Worth re-measuring under load, but not the dominant cost.

Direct mode becomes worth revisiting **if all three** are true: (a) chat-loop latency is measurably tight after v1, (b) nothing else needs MCP-via-gateway (just agentfarm), (c) we're willing to make agentfarm a critical platform component. Don't prematurely optimize for it.

### 5.8 Connection lifecycle:  bootstrap-then-stream

Polling on every call works but wastes a round-trip per call. Better pattern: **bootstrap on first use, upgrade to a persistent stream, close on idle.**

**Phase 1 — Cold start (no existing connection):**
1. agentfarm needs to call an MCP tool but has no active stream to the workspace's gateway.
2. agentfarm calls papi with the request; papi queues it and waits.
3. Gateway is doing its baseline outbound poll (or maintains a long-lived "control" stream) and picks up an `OPEN_STREAM` signal alongside the queued request.
4. Gateway opens a persistent bi-directional channel back to papi (WebSocket, HTTP/2 streaming, or gRPC bidi). The pending request is the first message on the new stream.

**Phase 2 — Steady state (stream is open):**
- agentfarm → papi → gateway: papi multiplexes incoming MCP requests onto the established stream.
- gateway → papi → agentfarm: responses flow back over the same stream.
- Latency per call is now: agentfarm↔papi RTT + papi↔gateway RTT (already-open socket) + gateway↔upstream MCP RTT. No polling overhead.

**Phase 3 — Close:**
The question of *when* to close matters. Three signals to choose from:

- **Idle timeout (recommended primary trigger):** close after N minutes of no traffic. For chat sessions, N = 5–10 min is reasonable — typical human pause between messages is well under that, idle gap longer than that usually means the session is over.
- **Explicit session-end signal:** when agentfarm's chat session ends, signal "no more calls expected" — close immediately. Avoids stranded connections.
- **Per-call expiry / TTL:** every persistent connection auto-closes after T minutes of total lifetime regardless of activity (e.g. T = 30 min). Keeps a hard upper bound on stale state.

Use **idle timeout + session-end + a 30-min hard cap**. Idle is the main signal; session-end is a fast cleanup; the hard cap is defense-in-depth.

**Reconnect behaviour:** after close, the next call falls back to Phase 1 cold start. Pays the bootstrap RTT once, then resumes streaming. Fine for the chat-loop pattern (user resumes after a coffee break — first message pays bootstrap cost, then is fast again).

**Keepalive:** ping every 30s while the stream is open to keep NATs/load-balancers from killing it.

**Cold start latency budget:** one outbound poll cycle. If the gateway maintains a baseline control-stream (always open), bootstrap is sub-100ms. If we go pure poll-based with a 5s poll interval, worst case bootstrap is 5s. **Recommendation:** the gateway maintains a lightweight always-on control channel for `OPEN_STREAM` and `HEARTBEAT` signals; data streams open on demand. Best of both — low cold-start latency, no idle data-stream cost.

### 5.9 Open questions for B

- **Protocol:** long-poll (simplest), gRPC bi-directional streaming (efficient, more infra), or WebSocket (mid-ground)? Each has clear trade-offs but needs a deliberate pick.
- **Gateway packaging:** sidecar in the runner pod, separate deployment in the same chart, or runner-binary subcommand? Sidecar is cleanest separation; subcommand is fewest moving parts.
- **Direct vs. via-papi:** §5.7 recommends via-papi. Confirm before building.
- **Connection lifecycle parameters:** idle timeout, hard cap, keepalive interval — pick concrete defaults.
- **Observability:** what's the agentic-loop debugging story when an MCP call fails? Logs in agentfarm, in papi, in the gateway, in the upstream MCP — need a correlated trace ID stitched across all four.
- **Failure modes:** if the gateway is down, how does agentfarm find out fast (so it doesn't hang the chat session)?

## 6. Approach C — 1 SLX per MCP server (tools-as-context)

A minimal SLX-as-proxy variant. Each registered MCP server gets one SLX with a generic codebundle. At chat-session start, agentfarm discovers the available tools and **injects them into the conversation context** (system prompt / initial assistant note) rather than registering each as an ADK callable. The LLM picks tools the same way it picks any other action — by reading the context and emitting a tool-call intent — which agentfarm routes through the existing task-run flow.

```
| bootstrap |                     +-----+-----+        | +-+---------+----------+-------+         |
| injects   |                           | reads        |   |3        |3         |3               |
| tools[]   |                           v              |   v         v          v                |
| into ctx  |                     +-----------+        | +-------+ +-------+ +--------+          |
+-----------+                     | MCPServer |        | | MCP#1 | | MCP#2 | |  MCP#3 |          |
                                  | table     |        | +-------+ +-------+ +--------+          |
                                  +-----------+        +------------------------------------------+

Steps: 1 agentfarm fetches server list, parallel-invokes list_tools on each server's SLX,
aggregates results, injects them into chat context. On LLM tool intent (2), invokes the
corresponding SLX with op=call_tool + tool_name + args. 3 SLX HTTP POSTs to local MCP.

Drift handling: on tool-not-found / schema-mismatch, agentfarm re-runs list_tools for that
server, refreshes context, retries once.  LLM trade-off: two surfaces for the LLM
(existing task-search + tools-in-context).
```

The codebundle is a single small script with two ops:
- `op = list_tools` → HTTP GET to the MCP server's `tools/list`, returns `[{name, description, inputSchema}, ...]`.
- `op = call_tool` → HTTP POST to the MCP server's `tools/call` with `{name, arguments}`, returns the result.

One SLX per registered MCP server, auto-created at registration time. Server URL + credentials baked into that SLX (workspace-secret scope keeps the credential blast radius tight).

### 6.2 Data flow

**Session bootstrap (once per chat session):**
1. Chat session begins.
2. agentfarm queries papi for the workspace's registered MCP servers.
3. agentfarm fires N parallel SLX invocations with `op=list_tools` (one per server).
4. agentfarm aggregates all returned tools into a single "Available MCP Tools" block — name, description, input schema, source server.
5. agentfarm **injects that block into the chat session's context** (system prompt or a leading assistant message). The LLM now knows the tools exist alongside its existing knowledge of RunWhen tasks.

**Per tool invocation:**
1. LLM reads the context, decides to use a tool, emits an intent — either as free-text or via a single generic ADK callable like `call_mcp_tool(server, tool_name, args)` the LLM is taught to use.
2. agentfarm parses the intent and invokes the corresponding server's SLX with `{op: "call_tool", tool_name, args}` via the existing task-run flow.
3. SLX runs, HTTP POSTs to the local MCP server's `tools/call`, returns the result.
4. Result flows back through the standard task-output channel and is appended to the chat context for the LLM to read.

### 6.3 Schema drift handling

If `call_tool` returns "tool not found" or a schema-mismatch error, agentfarm re-runs `list_tools` for that server, refreshes the tools block in context, and retries the call once. No cron, no platform-side cache, no TTL — drift is detected on failure and self-heals.

### 6.4 Required changes by repo

| Repo | Change |
|---|---|
| `rw-generic-codecollection` | One new generic codebundle (~100-200 lines): handles `list_tools` and `call_tool` ops. |
| `papi` | (a) `MCPServer` table per workspace `{server_id, name, url_ref, secret_ref, slx_name, ...}`; (b) auto-create-SLX flow on registration; (c) list-MCP-servers endpoint for agentfarm; (d) UI surface for registration. |
| `agentfarm` | Session-bootstrap discovery (parallel `list_tools` SLX runs + context injection) + a single generic dispatch path (one `call_mcp_tool` callable, or output-parsing) + drift-on-failure refresh. No per-tool ADK registration. |
| `runwhen-runner` | None. |
| `platform-robot-runtime` | None. |

### 6.5 Pros

- **Very small implementation.** Codebundle + papi CRUD/registry + agentfarm bootstrap + context injection. Smallest of all four approaches.
- **Reuses everything.** SLX run primitive, secret management, output channel, audit, scheduling — all unchanged.
- **Schema drift handled implicitly.** No cron, no platform-side sync, no cache.
- **Few SLXs per workspace.** One per registered server (typically 3-10 total). Low UI clutter.
- **Zero new customer-side deployment.**
- **Each MCP call is auditable as an SLX run.**

### 6.6 Cons / risks

- **Two surfaces for the LLM to reason about.** Existing task-search workflow + MCP tools in context. The LLM has to pick from both. *This is the main reason to consider Approach D instead.*
- **Tools-as-context is "soft."** Less structured than first-class tasks (or ADK callables) — LLM may hallucinate tool names or args. Mitigated by clear prompting + drift retry, but not eliminated.
- **Per-tool RBAC / audit / search-visibility are not first-class.** Tools are ephemeral context, not platform entities.
- **Context bloat risk.** A server with 50 tools × 5 servers = lots of tokens in every session's context. Mitigations: only inject summaries; on-demand tool detail fetch.
- **SLX-run latency floor** (1-15s) applies to every tool call. Same as A and D — acceptable for low/medium volume; not for tight agentic loops.

### 6.7 Open questions for C

- **Tool-call interface.** Single generic `call_mcp_tool(server, tool_name, args)` ADK callable, or free-form LLM output parsing, or some hybrid? Choice affects reliability.
- **Context refresh policy.** Inject once at session start, or refresh per turn / on user "new question" signal?
- **Tools-block size.** Full schemas in context vs. summaries with on-demand detail fetch.
- **MCP transport.** v1 = HTTP/JSON-RPC only.
- **SLX cleanup on deregistration** — delete, soft-delete, or archive.

## 7. Approach D — 1 SLX per MCP tool (tools-as-tasks)   [recommended]

The most fully integrated variant. **Every MCP tool becomes a first-class RunWhen task** (its own SLX). Per-tool SLXs are **generated by the runner-side workspace builder** using a standard generation rule + templates — the same mechanism the codecollection already uses to auto-generate SLXs from discovered Kubernetes namespaces, GCP functions, Azure VMs, etc. agentfarm's existing task-search and task-run workflow handles MCP tools with **zero new agent-side code** — they're just more tasks in the workspace.

**Key insight:** the sync mechanism we don't want to build on the platform side **already exists as the workspace-builder cycle on the runner.** No new cron in papi, no new reconcile logic — reuse the existing builder pattern.

### 7.1 Architecture

**Diagram:** [`approach-d-slx-per-tool.excalidraw`](./approach-d-slx-per-tool.excalidraw) — open with [excalidraw.com](https://excalidraw.com) or the VS Code / Obsidian Excalidraw extension.

```
                Platform side                        │       Customer VPC

[agentfarm]                                          │

     v                                               │

     │           (URL, secret_ref, display_name)     │           │
     │                                               │           │ for each registered MCP server:
     │                                               │           │   - call tools/list (in-VPC)
     │                                               │           │   - emit "mcp_tool" resources
     │                                               │           │
     │                                               │           v
     │                                               │   [Generation Rule]  (mcp-tool-proxy/.runwhen/...)
     │                                               │           │

     │                                               │           │ from templates  (constant codebundle)
     │                                               │           v
```

**Two parts:**

1. **Generation side (runner workspace-builder):** a generation rule + templates in a new codebundle directory `codebundles/mcp-tool-proxy/.runwhen/`. The builder reads registered MCP servers from papi, calls each server's `tools/list` (in-VPC), produces `mcp_tool` resources, the generation rule renders one SLX per resource.

2. **Execution side (codebundle script):** a constant Python script in `codebundles/mcp-tool-proxy/` that does the MCP `tools/call` for the SLX's baked-in `MCP_TOOL_NAME`. Reads runtime variables for tool args. Pattern follows the reference script provided (init → notifications/initialized → tools/call → return result).

### 7.2 Data flow

**Builder cycle (existing workspace-builder cadence in the runner):**
1. Builder runs (periodically, on existing schedule).
2. Builder pulls the workspace's registered MCP server list from papi (one outbound API call).
3. For each registered server: in-VPC call to `tools/list`. Returns the current tool catalog with names/descriptions/input_schemas.
4. Each tool emits an `mcp_tool` resource: `{server_id, server_url, secret_ref, tool_name, description, input_schema}`.
5. The `mcp-tool-proxy` generation rule matches these resources and renders one SLX per tool from the templates — name = `mcp__{server}__{tool}`, alias = tool description, configProvided = `{MCP_SERVER_URL, MCP_TOOL_NAME}`, secretsProvided = the server's auth secret reference.
6. Builder's normal reconcile pass: create new SLXs, update changed ones, remove SLXs for tools that no longer exist.

**Per tool invocation (existing workflow, untouched):**
1. User asks something.
2. agentfarm's existing task-search ranks workspace SLXs (including MCP-derived ones) against the query.
3. agentfarm invokes the matching SLX with user-supplied args via existing runtime variables → existing task-run flow.
4. SLX runs the constant codebundle script which HTTP POSTs to the MCP server's `tools/call`.
5. Result flows back via standard task-output channel; agentfarm feeds it to the LLM.

### 7.3 Schema drift handling

**The workspace-builder cycle is the drift handler.** Every builder run re-introspects, re-renders, and reconciles. No separate sync system to design.

- **Cadence:** whatever the workspace builder runs at today (no new schedule).
- **On-registration:** trigger a builder run immediately (or wait for the next scheduled cycle, depending on UX preference). Existing workspace-builder triggers may already cover this.
- **On-failure (optional):** agentfarm could signal "tool not found" → trigger an early builder run for that workspace.

### 7.4 MCP server discovery — Option D1 vs. Option D2

The workspace-builder needs to know *which* MCP servers to introspect. Two viable mechanisms; both end with the same result (`mcp_tool` resources for the generation rule to consume) but differ in *where* the registration lives.

#### Option D1 — workspace-builder pulls from papi

The customer registers an MCP server through the platform UI; papi stores it in the `MCPServer` table; the workspace-builder makes an outbound API call to papi at the start of each cycle to retrieve the current list, then introspects each one.

```
Customer UI ─register──> [papi: MCPServer table] <───GET /workspaces/{ws}/mcp-servers─── [workspace-builder]
                                                                                              │
                                                                          ┌───────────────────┘
                                                                          v
                                                                   for each server:
                                                                     in-VPC POST tools/list
                                                                          │
                                                                          v
                                                                   emit mcp_tool resources
```

**Pros:**
- Single source of truth (papi).
- Customer manages MCP servers via the platform UI like any other workspace resource. Adding/removing/updating an MCP server takes effect on the next builder cycle — no Helm upgrade needed.
- Centralized governance: registrations are visible to platform admins, auditable, integrate with workspace RBAC.
- Reuses existing platform-→-runner API pattern. Runner already makes outbound calls to papi for other state; this is just one more.
- Secret references live in the platform's existing workspace-secret vault.

**Cons:**
- Workspace-builder gains a dependency on papi being reachable at cycle start. Existing SLXs persist if a single cycle fails, so the failure mode is "tools-list doesn't refresh," not "everything breaks."
- Slight latency at the start of each cycle (one extra outbound API call). Negligible.
- Requires papi-side work: `MCPServer` table + CRUD endpoints + UI form.

#### Option D2 — define in workspace-builder Helm values

MCP servers are declared in the runner's Helm chart values (or a ConfigMap). The workspace-builder reads the config at startup and during cycles. Adding/removing a server requires a Helm upgrade.

```yaml
# values.yaml  (runner Helm chart)
mcpServers:
  - name: jira
    displayName: "Internal Jira"
    url: "https://jira-mcp.internal:443/mcp"
    secretRef: "jira-mcp-token"   # references existing workspace-secret
  - name: linear
    displayName: "Linear"
    url: "https://linear-mcp.internal:443/mcp"
    secretRef: "linear-mcp-token"
```

**Pros:**
- No platform-side API dependency for the discovery path. Builder is fully self-sufficient.
- GitOps-friendly: MCP server config lives alongside other infra config; reviewed and rolled out via the customer's normal Helm-upgrade process.
- Reduces v1 scope on the platform side — no `MCPServer` table or registration UI needed up front.
- Customer ops teams that prefer file-based config get a familiar workflow.

**Cons:**
- Adding/removing an MCP server is a Helm operation (chart values change + redeploy). Friction for non-ops users.
- No platform UI for end users — admins have to know what's registered by reading values.yaml.
- Cross-customer visibility is poor: platform admins can't see what each workspace has registered without inspecting each runner's config.
- Secret references still need to point at workspace secrets (or be Helm-managed), so the secret-management story is split between two systems.
- Schema drift on the *registration* itself (e.g., the URL changes): customer must trigger a Helm upgrade for the builder to pick it up.

#### Recommendation: **D1 (pull from papi)** for v1

D1 is the better long-term shape:
- Better UX for the people actually using the platform.
- Aligns with how every other workspace resource is managed (UI → papi → runner via API).
- Avoids fragmenting MCP-server config between Helm and the platform.
- The papi-side work is small and additive (one table + one endpoint + one form).

**D2 as a v0 / prototype path:** if the team wants to ship a proof-of-concept faster without doing the papi UI work, D2 is a valid intermediate. The migration story is straightforward: once papi support lands, the Helm-based config falls back to "no registered servers," and customers re-register via the UI.

**Could we support both?** Yes — the builder could read both sources and merge them. Not worth the complexity for v1; pick one and grow if there's real demand for the other.

#### 7.5.6 Runbook template  (`.runwhen/templates/mcp-tool-proxy-runbook.yaml`)

```yaml
apiVersion: runwhen.com/v1
kind: Runbook
metadata:
  name: {{slx_name}}
spec:
  location: {{default_location}}
  codeBundle:
    repoUrl: https://github.com/runwhen-contrib/rw-generic-codecollection.git
    ref: main
    pathToRobot: codebundles/mcp-tool-proxy/runbook.robot
  # --- Static, per-SLX -----------------------------------------------------
  # Same for every invocation. Server URL and tool name identify *which* MCP
  # tool this SLX proxies; the full input schema is shipped along so the
  # codebundle can introspect what parameters to expect at runtime.
  configProvided:
    - name: MCP_SERVER_URL
      value: {{match_resource.spec.server_url}}
    - name: MCP_TOOL_NAME
      value: {{match_resource.spec.tool_name}}
    - name: MCP_INPUT_SCHEMA
      value: '{{match_resource.spec.input_schema | tojson}}'
  secretsProvided:
    - name: mcp_auth
      workspaceKey: {{match_resource.spec.secret_ref}}
  # --- Dynamic, per-invocation ---------------------------------------------
  # One entry per MCP tool input parameter, rendered from the tool's JSON
  # schema. These are supplied by agentfarm at task-run time (mapped from the
  # LLM's tool-call arguments) and merged into the runner env by papi — see
  # the callout below for the merge mechanism.
  runtimeVarsProvided:
    {% for pname, pschema in match_resource.spec.input_schema.properties.items() %}
    - name: {{pname}}
      type: {{pschema.type | default("string")}}
      description: "{{pschema.description | default('') | replace('"','\\"')}}"
      {% if pschema.default is defined %}default: {{pschema.default | tojson}}{% endif %}
      required: {{pname in (match_resource.spec.input_schema.required | default([]))}}
    {% endfor %}
```

> **Why this works with no platform-side changes**
>
> papi's `assemble_runbook_env` (`backend-services-v2/papi/routers/explorer.py:496-547`) accepts arbitrary `runtime_var_values: dict[str, str]` from the RunRequest payload and overlays them onto `configProvided` with last-write-wins semantics — there's no allowlist or pre-declaration check at this layer. So whatever per-parameter values agentfarm packages into the run request (`{"project": "ENG", "summary": "..."}`) land in the runner's process environment under those exact names. The codebundle picks them up via `RW.Core.Import User Variable ${pname}` inside a loop over the schema. **No new field on the Runbook CRD, no new code path in papi or runner-control.**
>
> Two bonuses fall out of this:
> - **Defaults & validation** declared in `runtimeVarsProvided` are merged in upstream by `validate_runtime_var_overrides` at RunRequest creation, so the codebundle doesn't need to re-implement type coercion or required-field checks.
> - **Tool-builder `CONFIG_ENV_MAP` auto-extension** (lines 527-547 of the same file) means even bash-subshell-style codebundles get runtime vars passed through without authors mirroring declarations in two places. Our MCP wrapper runs Python directly so it doesn't need this, but it's worth knowing the mechanism exists.

**Python script outline** (`mcp_tool_proxy.py`) — pure MCP client. Reads the packaged tool-args blob the Robot wrapper assembled, runs the standard initialize → `tools/call` handshake, returns the tool's response as plain text:

```python
def main():
    server_url = os.environ["MCP_SERVER_URL"]
    tool_name  = os.environ["MCP_TOOL_NAME"]
    tool_args  = json.loads(os.environ.get("MCP_TOOL_ARGS_JSON", "{}"))
    auth_token = os.environ["MCP_AUTH"]

    session = requests.Session()
    session.headers.update({
        "Content-Type": "application/json",
        "Authorization": f"Bearer {auth_token}",
        "Accept": "application/json, text/event-stream",
    })

    # 1. initialize MCP session
    init = _rpc(session, server_url, "initialize", {
        "protocolVersion": "2025-03-26",
        "capabilities": {},
        "clientInfo": {"name": "runwhen-mcp-proxy", "version": "1.0.0"},
    }, request_id=1)
    sid = init.get("session_id")
    if sid:
        session.headers["Mcp-Session-Id"] = sid
    _notify(session, server_url, "notifications/initialized")

    # 2. invoke the tool with the args the Robot wrapper packaged from runtime vars
    result = _rpc(session, server_url, "tools/call",
                  {"name": tool_name, "arguments": tool_args}, request_id=2)

    # 3. extract the tool's response content (MCP returns result.content as an array
    #    of typed parts: text, image, resource, etc.). For tool-calls-only v1, we
    #    expect text parts; collapse to a single string for task output.
    content_parts = (result.get("result", {}) or {}).get("content", [])
    text_chunks = [p["text"] if p.get("type") == "text"
                   else json.dumps(p, indent=2)
                   for p in content_parts]
    tool_output = "\n".join(text_chunks) if text_chunks else json.dumps(result.get("result", {}), indent=2)

    # 4. write tool output to stdout — Robot wrapper captures and surfaces via
    #    RW.Core.Add Pre To Report. NOT wrapped as an issue.
    print(tool_output)
```

Failure handling: if MCP init or `tools/call` returns an error, write the error to the task's stderr and exit non-zero so the task is marked failed. agentfarm sees a failed task and can react (retry, surface to user) using its existing failure path. Only emit an issue in the unusual case where surfacing the failure to the workspace's issue feed adds value — for the default "tool returned an error" case, a failed task is enough.

| `rw-generic-codecollection` | New `codebundles/mcp-tool-proxy/` containing: (a) constant execution script (Python — does MCP init + `tools/call`, pattern from the reference script); (b) `runbook.robot` wrapping the script; (c) `.runwhen/generation-rules/mcp-tool-proxy.yaml` matching `mcp_tool` resources; (d) `.runwhen/templates/` rendering SLX + Runbook (+ optional SLI). |

| `runwhen-runner` / workspace-builder | New `mcp_tool` resource discovery probe — calls registered MCP servers' `tools/list`, emits resources for the generation rule to consume. Depending on builder extensibility, this might just be a small script invoked as part of the existing discovery phase. |
| `platform-robot-runtime` | None. |

### 7.7 Pros

- **Zero agentfarm changes.** Strongest single argument for D. Agent already knows how to search and run workspace tasks.
- **Sync mechanism is free.** Reuses the existing workspace-builder cycle. **No new cron, no new reconcile, no new platform-side state.**
- **Same generation-rule pattern** every other codebundle in the codecollection already uses. Familiar to anyone who's added a generation rule before. No new patterns to learn.
- **Unified LLM mental model.** Every callable thing is a task. No second surface.
- **MCP tools as first-class platform entities.** Per-tool RBAC, audit, search ranking, scheduling — all extend for free.
- **Workspace UI shows MCP tools as discoverable tasks** — humans benefit too.
- **Per-tool credentials scoped tight.** Each per-tool SLX inherits its server's secret reference via the template.

### 7.8 Cons / risks

- **Workspace-builder cycle is the discovery cadence.** Between builder runs, a newly-added MCP tool isn't visible. Mitigated by short cycle + optional on-registration builder trigger + optional on-failure trigger.
- **Workspace-builder needs to support a new probe type** (or a script-driven discovery). Depending on how extensible the builder is, this is small or medium effort.
- **More SLXs per workspace** (N servers × M tools). UI clutter risk mitigated by tags/grouping.
- **Tool-rename / delete handling** depends on workspace-builder reconcile semantics — likely already handles add/update/remove for other resource types; reuse the same path.
- **SLX-run latency floor** (1-15s) applies — same as A and C.
- **MCP semantics ceiling unchanged.** Tool calls only.

### 7.9 Open questions for D

- **D1 vs. D2 discovery.** §7.4 recommends D1 (pull from papi). Confirm with the team before implementation.
- **Probe failure semantics.** If `tools/list` fails for one server during a builder cycle, do we keep the previous SLXs (preserve state) or remove them (treat as gone)? Probably preserve to avoid flap.
- **SLX naming.** `mcp__{server_name}__{tool_name}` proposed; confirm fits existing SLX-naming conventions.
- **UI grouping / filtering** of MCP-derived SLXs for visual cleanliness — show grouped by server? Hide vs. show by default?
- **Per-tool RBAC defaults** for auto-created SLXs — inherit from registering user, server-level policy, workspace default?
- **Initial-registration UX.** Wait for next builder cycle (could be minutes) vs. immediately trigger a one-shot builder run? Latter is better UX but more plumbing.
- **MCP transport.** v1 = HTTP/JSON-RPC only. stdio / SSE out of scope.
- ~~**Input-schema → runtime-variable mapping.** How does the platform present the MCP tool's input schema to agentfarm so it sends the right args?~~ **Resolved.** The generation rule emits one `runtimeVarsProvided` entry per `input_schema.properties` field (see §7.5.6). Agentfarm consumes the SLX's declared runtime vars exactly the way it does for any other task today, and papi's `assemble_runbook_env` merges them into the runner env at invocation time (`explorer.py:496-547`). The `MCP_INPUT_SCHEMA` static config var is kept as a redundant hint for the codebundle's own dynamic-import loop.

### 8.1 On latency specifically

Concrete shape per-call:

- **A / C / D (SLX-run path):** `agentfarm → papi → worker queue → runner poll → robot.run() boot → suite init → secret fetch → script exec → HTTP call to local MCP → return`. Baseline 5-15s cold; achievable 1-3s with priority worker class + warm caches + pre-pooled codebundle pods.
- **B (gateway path):** `agentfarm → papi MCP router → match to gateway stream → MCP call from gateway → return`. 100ms-1s baseline.

For low/medium tool-call volume in chat sessions, the SLX-run floor is acceptable — users routinely wait that long for LLM tokens to stream. For high-frequency or tight agentic loops, the floor would hurt.

### 8.2 On effort specifically

- **A** spreads changes across load-bearing existing systems (Runbook model, its consumers, codebundle, possibly robot-runtime), plus introduces a cron resync flow.
- **B** concentrates changes in one new component (the gateway) plus a contained new feature in papi (the router) — but that "new component" includes deploy chart, outbound protocol, auth bridge, observability infra, all of which are months of engineering for a service category the platform doesn't currently have.
- **C** is the smallest in raw line count: codebundle + papi CRUD + agentfarm bootstrap and dispatch. Trade-off is two LLM surfaces (existing tasks + MCP-tools-in-context).
- **D** has *zero* agentfarm changes and *no new sync mechanism* — the SLX generation is owned by the existing runner-side workspace builder using the standard generation-rule + templates pattern that every other codebundle in the collection already uses. **Lowest end-to-end integration cost of all four.**

When the only meaningful agentfarm-side cost is reduced to nothing AND the sync mechanism is reused from existing infrastructure, D becomes clearly the smallest investment for the most integrated outcome.

### 8.3 C vs D — when each one fits

C and D both use the SLX-as-proxy model, but the choice between them is essentially about **where the integration cost lives** and **how MCP tools surface to the LLM**.

| Question | C wins | D wins |
|---|---|---|
| Where's the integration cost easier to absorb? | agentfarm team has bandwidth, papi sync flow is undesirable | papi sync flow is acceptable, agentfarm changes are expensive |
| How should MCP tools surface to the LLM? | As ephemeral context (lighter, less structured) | As first-class searchable tasks (heavier, more structured) |
| What's the typical tool count per workspace? | Many (50+ per server) — context injection is the only sane option | Modest (5-30 per server) — every tool as a task is fine |
| Do tools need per-tool RBAC, audit, history? | No, ephemeral usage is fine | Yes, treat them like any other task |
| Is sync infrastructure (cron + reconcile) a problem? | Yes — drift-on-failure is much simpler | No — sync is acceptable engineering |

## 9. Recommendation

**Build Approach D — 1 SLX per MCP tool (tools-as-tasks).**

The journey: we initially leaned B (gateway) after rejecting A (multi-task SLX) on blast-radius grounds. Red-teaming both surfaced a simpler SLX-as-proxy variant (C). Then the realization that *every MCP tool can just be a regular RunWhen task* produced D — which has the lowest end-to-end integration cost and the cleanest LLM surface of all four.

Rationale:

1. **Zero agentfarm changes.** The agent already has a task-search-and-run workflow; MCP tools as tasks slot in with no special-case code.
2. **No new sync infrastructure.** Reuses the runner-side workspace-builder cycle and the standard generation-rule + templates pattern. **The sync mechanism we'd otherwise build in papi already exists in the runner.**
3. **One unified LLM surface.** No "tools in context" + "tasks via search" bifurcation. The LLM has one mental model.
4. **MCP tools as first-class platform entities.** Per-tool RBAC, audit, search ranking, scheduling, runbook composition — all extend for free.
5. **Customer UI shows MCP tools as discoverable tasks** — humans benefit too.
6. **Time-to-build is bounded and small.** Codebundle (constant script + generation rule + templates) + papi CRUD/registry + UI registration. Pattern is familiar to anyone who's added a codecollection generation rule.
7. **Honest about the ceiling.** Tool calls only. Same as A and C. Revisit B if MCP scope expands or volume grows high.
8. **Multi-task SLX work stays decoupled.** Build it later for tool-builder users on its own merits if you want it. MCP doesn't force it.

### When Approach C would be the right call (over D)

- A typical MCP server exposes 50+ tools and you don't want hundreds of auto-managed SLXs per workspace cluttering the UI.
- agentfarm team has bandwidth for a small bootstrap-and-dispatch integration but the platform team strongly prefers to avoid the sync mechanism.
- Self-healing drift (no platform-side sync) is more valuable than per-tool first-class status.

### When Approach B would be the right call (over D)

- High tool-call frequency where the SLX-run latency floor (1-3s hot, 5-15s cold) is too slow for tight agentic loops.
- Real customer demand for MCP semantics beyond tool calls (resources, prompts, sampling, streaming progress).
- Platform-engineering capacity to own a new networked-service category long-term.

### When Approach A would be the right call (over D)

- You want multi-task SLX for non-MCP reasons (tool-builder users want it). MCP becomes a downstream beneficiary of work justified on its own merits.

### When to revisit D → B

- Tool-call volume hits ~100 calls/min/customer → latency floor of D will hurt.
- MCP scope expands beyond tool calls → D cannot extend cleanly.
- D ships, runs in production, generates usage data — that data tells you whether B is actually needed.

## 13. Appendix A — Reference MCP-client pattern (Linear)

The execution-side Python script in `codebundles/mcp-tool-proxy/mcp_tool_proxy.py` will follow the **MCP-protocol exchange pattern** shown below — a prototype the user built against Linear's hosted MCP. **Differences in the production codebundle:**

- Tool name comes from `MCP_TOOL_NAME` env (set by the generation-rendered SLX), not user input.
- Args come from `RW_INVOCATION_ARGS_JSON` (runtime variables from the task invocation), not the `MCP_TOOL_PARAMS` env.
- Only the `tools/call` path is exercised at runtime — the `tools/list` path lives in the workspace-builder probe, not the per-tool SLX.
- **Output is exposed as task output, not as an issue.** The reference below wraps everything in the `issue title / issue description / issue severity / issue next steps` shape because it predates this design and the user was reusing the tool-builder runbook scaffolding. The production codebundle returns plain tool output (`RW.Core.Add Pre To Report` or equivalent) — the MCP response IS the answer, not a problem report. See §7.5.7 for the production-shape script outline.

```python
import os, json, requests

def _read_secret(env_var):
    """RunWhen injects secrets as file paths; read the file content."""
    path = os.environ.get(env_var, "")
    if not path:
        return ""
    if os.path.isfile(path):
        with open(path, "r") as f:
            return f.read().strip()
    return path


def _parse_response(resp):
    """Parse JSON or SSE response from MCP server."""
    ct = resp.headers.get("Content-Type", "")
    if "text/event-stream" in ct:
        for line in resp.text.split("\n"):
            if line.startswith("data: "):
                try:
                    msg = json.loads(line[6:])
                    if "id" in msg or "result" in msg:
                        return msg
                except json.JSONDecodeError:
                    continue
        return None
    try:
        return resp.json()
    except Exception:
        return None


def main():
    token       = _read_secret("LINEAR_API_TOKEN")
    tool_name   = os.environ.get("MCP_TOOL_NAME", "list_tools")
    tool_params = json.loads(os.environ.get("MCP_TOOL_PARAMS", "{}"))
    mcp_url     = "https://mcp.linear.app/mcp"

    session = requests.Session()
    session.headers.update({
        "Content-Type":  "application/json",
        "Authorization": f"Bearer {token}",
        "Accept":        "application/json, text/event-stream",
    })

    # initialize MCP session
    init_resp = session.post(mcp_url, json={
        "jsonrpc": "2.0", "id": 1, "method": "initialize",
        "params": {
            "protocolVersion": "2025-03-26",
            "capabilities": {},
            "clientInfo": {"name": "runwhen-mcp-client", "version": "1.0.0"},
        },
    }, timeout=30)
    init_result = _parse_response(init_resp)
    sid = init_resp.headers.get("Mcp-Session-Id")
    if sid:
        session.headers["Mcp-Session-Id"] = sid
    session.post(mcp_url, json={"jsonrpc": "2.0",
                                "method": "notifications/initialized"}, timeout=30)

    # call the tool
    call_resp = session.post(mcp_url, json={
        "jsonrpc": "2.0", "id": 2, "method": "tools/call",
        "params": {"name": tool_name, "arguments": tool_params},
    }, timeout=30)
    call_result = _parse_response(call_resp)

    # convert MCP result into the RunWhen issue/output format used by tool-builder
    content = call_result.get("result", {}).get("content", [])
    parts = [item["text"] if item.get("type") == "text"
             else json.dumps(item, indent=2)
             for item in content]
    output = "\n".join(parts) if parts else json.dumps(call_result.get("result", {}), indent=2)

    return [{
        "issue title":       f"MCP: {tool_name}",
        "issue description": output[:3000],
        "issue severity":    4,
        "issue next steps":  "Review tool output above.",
    }]
```

The generalized codebundle version replaces the hard-coded `mcp_url` with `os.environ["MCP_SERVER_URL"]`, replaces the hard-coded secret env name with `mcp_auth` (matching the template's `secretsProvided`), and drops the `list_tools` branch (handled in the builder probe instead).
